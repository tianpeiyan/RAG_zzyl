# 基于 LangChain 的养老系统 RAG 智能助手 V1 计划

## Summary
在现有项目上新增一个“知识问答型智能助手”子系统，首版聚焦 `知识库问答 + 系统功能导航`，暂不接入真实老人档案、健康信息和护理业务数据。

整体采用 `Vue3 管理端 + Java 网关/BFF + Python LangChain 服务` 架构：
- 前端只调用现有 Java `/api` 接口。
- Java 负责登录态复用、权限校验、审计记录、转发 Python AI 请求。
- Python 服务负责文档解析、切片、Embedding、向量检索、提示词编排和回答生成。
- 模型与 Embedding 默认使用 `阿里百炼/通义`，向量库首版使用 `Chroma` 持久化存储。

## Key Changes
### 1. AI 服务子系统
新增独立 Python 服务，职责固定为知识库问答，不直接访问业务库。
- 提供内部接口：聊天问答、文档导入、重建索引、索引状态查询、文档列表查询。
- 使用 LangChain 编排：`文档加载 -> 清洗切片 -> Embedding -> Chroma 检索 -> LLM 生成`。
- 首版按中文场景优化，提示词约束为：
  - 仅依据知识库内容回答。
  - 不确定时明确说“知识库暂无依据”。
  - 对系统导航类问题优先输出“菜单路径 + 操作步骤”。
- 检索策略默认采用 `向量检索 + 基础关键词召回兜底`，命中不足时拒答，不做自由发挥。
- 虽然前端首版不展示出处，但内部保留 `命中文档、分片、分数、回答链路日志`，为后续溯源做准备。

### 2. Java 集成层
在现有 Spring Boot 后端新增助手代理与管理接口，统一挂在 `/api/assistant/*`。
- 对外聊天接口：接收前端问题、会话信息，校验登录态后转发 Python 服务。
- 对外知识库管理接口：文档列表、索引状态、手动重建索引、失败任务查看。
- Java 到 Python 的调用携带内部服务身份和当前用户上下文，用于审计，不把前端 token 直接暴露给 Python。
- 记录最小审计日志：提问人、时间、问题摘要、回答状态、命中文档数量、失败原因。
- 首版不做细粒度业务权限判断，因为不接真实老人数据；只保留后台菜单权限控制。

建议新增的公开接口形态：
- `POST /api/assistant/chat`
- `GET /api/assistant/sessions/{id}` 或简化为前端本地会话
- `GET /api/assistant/kb/documents`
- `GET /api/assistant/kb/index/status`
- `POST /api/assistant/kb/index/rebuild`

### 3. 知识库与运营方式
知识源采用“固定目录导入 + 基础管理页”模式。
- 约定知识目录，例如 `knowledge-base/`，按类别分子目录：
  - `system-manuals/`
  - `care-specs/`
  - `process-guides/`
  - `faq/`
- 首版支持常见文件：`txt`。
- 导入流程由 Python 扫描目录并生成索引；管理页只负责查看文档状态和触发重建，不负责文件上传。
- 文档元数据至少包含：标题、分类、来源文件名、更新时间、索引状态、切片数量。
- 对文档做基础清洗规则：
  - 去除重复页眉页脚
  - 保留标题层级
  - 长表格和图片说明转为可检索文本
  - 大段空白和乱码直接丢弃
- 为后续扩展预留分类过滤能力，但 V1 默认全库检索。

### 4. 前端智能助手页面
在现有 Vue3 管理端新增独立“智能助手”页面，不做全局悬浮窗。
- 新增菜单入口，例如一级或二级菜单 `智能助手`。
- 页面包含：对话区、推荐问题、会话历史、回答状态、重试入口。
- 首版回答展示不强制显示引用来源，但需保留“该回答基于知识库生成”的提示。
- 对系统导航问题，前端支持把回答中的菜单路径做成可读格式，例如“服务管理 > 护理计划 > 护理等级”。
- 管理页单独提供基础运营能力：文档列表、索引状态、手动重建、失败提示。
- 首版不做语音输入、多轮工具调用、文档上传拖拽、答案评价闭环。

## Public APIs / Interfaces
需要明确并固定的接口与配置：
- Java 对前端公开 `/api/assistant/*`，前端不直连 Python。
- Python 对 Java 提供内部接口：
  - `POST /internal/chat`
  - `POST /internal/kb/reindex`
  - `GET /internal/kb/documents`
  - `GET /internal/kb/status`
- 新增 AI 配置项：
  - 百炼 API Key / 模型名 / Embedding 模型名
  - 知识目录路径
  - Chroma 持久化目录
  - 分片大小、重叠长度、召回条数、最低分阈值
- 前端新增页面路由与菜单资源：
  - 智能助手问答页
  - 知识库管理页

## Test Plan
核心验收场景：
- 输入护理规范问题，能命中知识库并输出稳定中文答案。
- 输入“某功能在哪里/怎么操作”，能回答到现有系统菜单路径与步骤。
- 输入知识库中不存在的问题，系统明确拒答，不编造。
- 知识目录新增或替换文档后，可通过管理页触发重建索引并成功生效。
- 非登录状态或无菜单权限用户不能访问助手页面或管理页。
- Python 服务异常、模型超时、索引缺失时，Java 和前端能给出可理解错误提示。
- 文档含重复内容、空白页、超长段落时，导入流程不崩溃且会输出失败/跳过记录。

建议测试组成：
- Python 单测：文档切片、检索链、提示词输出约束、拒答逻辑。
- Java 集成测试：代理接口、鉴权、异常转换、审计落库/落日志。
- 前端页面测试：对话发送、空态、加载态、失败重试、管理页状态刷新。
- 人工验收集：准备 20 到 30 条养老业务问题，覆盖制度、护理、系统导航、未命中问题。

## Assumptions
- V1 不接真实老人信息、健康档案、护理任务等结构化业务查询。
- V1 重点做知识型问答；流程指导通过知识库文档回答，不做工作流自动办理。
- 模型默认使用通义，向量库默认使用 Chroma；后续如需切换 Milvus/OpenAI，只替换 AI 服务适配层。
- 知识文档由业务方提前准备为“现成文档包”，项目组负责目录规范、解析和索引。
- 管理页首版只做“查看与重建”，不做上传、审核、版本回滚。
- 前端首版不展示出处，但后端必须保存命中文档元数据，避免后续返工。
